## Slide 02: 痛点与机会

**标题**: 个人理财的三大断层

### 痛点1: 数据采集断层
**现象**：

- 用户日常消费记录在支付宝、微信支付截图中
- 传统OCR对APP截图识别率接近0%（PaddleOCR实测）
- 用户被迫手动录入，5-10分钟/张账单
- 繁琐导致放弃使用

**真实场景**：
大学生拍了20张支付宝账单截图，想整理月度消费，但手动录入需要2小时。

### 痛点2: 决策信任断层
**现象**：
- 银行APP推荐基金/理财产品，只有结果没有推理
- 用户无法判断推荐是否符合自己的风险偏好
- 黑盒决策导致信任度低、转化率差

**真实场景**：
用户想攒钱买车，但不知道该选货币基金还是债券基金，银行推荐了混合型基金但不解释原因。

### 痛点3: 风险感知断层
**现象**：
- 异常支出（深夜大额消费、异地刷卡）总是事后发现
- 被动等待银行通知，缺乏主动预警机制
- 发现时已造成损失

**真实场景**：
信用卡被盗刷，深夜消费1200元，用户第二天早上才发现。

### 技术机会窗口

**Why Now?**
1. **技术成熟**：GPT-4o Vision（2024年）能直接理解复杂UI界面
2. **监管要求**：金融推荐需要可解释性（XAI），银行面临合规压力
3. **用户觉醒**：Z世代更重视数据隐私和决策透明度

**市场空间**（保守估计）：
- TAM：中国个人理财APP用户约3000万（艾瑞咨询2024）
- SAM：对隐私敏感 + 需要专业建议的年轻用户约300万
- SOM：首年目标1万付费用户（0.3%渗透率）

## Slide 03: 解决方案

**标题**: 三大技术创新重构理财体验

### 整体架构
**一步识别 + 透明推荐 + 主动预警**

### 创新1: Vision LLM一步提取
**传统方案的问题**：
```
图片 → OCR提取文字 → GPT结构化 → 结果
      ↑ 信息丢失        ↑ 上下文不完整
```

**我们的突破**：
```
图片 → GPT-4o Vision → 结构化数据
      ↑ 一步完成，保留完整上下文
```

**验证结果**：
- 测试数据：10张真实账单图片（72笔交易）
- 识别成功率：72/72（100%准确率）
- 处理速度：3秒/张（vs 手动录入5-10分钟）

### 创新2: Prompt Engineering解决多行识别
**问题诊断**：
- 初期测试：LLM只识别账单第一行（30%成功率）
- 原因分析：不是token限制，而是LLM理解问题
- 症状：bill_dining.png（4笔交易）只识别出1笔

**技术突破**：
改变LLM的思考流程 - "先统计数量，再逐行提取"

关键Prompt片段：
```
先统计图片中有多少笔交易（几行独立金额）
然后逐行提取每一笔的详细信息
确保 transactions 数组长度 = transaction_count
```

**突破效果**（2025-11-15验证）：
- bill_dining.png：1/4 → 4/4（100%）
- bill_shopping.png：1/3 → 3/3（100%）
- 10张测试图片：整体成功率30% → 100%

**技术本质**：
不是优化算法复杂度，而是**重构数据处理流程**，让LLM按正确的顺序思考。

### 创新3: XAI可解释推荐架构
**设计理念**：
规则引擎（合规层）+ LLM翻译层（理解层）

**架构图**：

```
用户输入（目标、风险偏好）
    ↓
规则引擎（风险评估、资产配置算法）
    ↓
推荐结果 + 规则参数
    ↓
LLM翻译层（转换为自然语言解释）
    ↓
用户看到：推荐 + 完整推理链
```

**核心优势**：
1. **可审计**：规则引擎的决策逻辑可追溯
2. **监管友好**：满足金融推荐的合规要求
3. **用户理解**：技术术语翻译成人话

## Slide 05: 技术架构（加上图片）

**标题**: 核心技术决策与工程实现

### 架构决策1: 放弃PaddleOCR，选择GPT-4o Vision

**决策背景**：
- PaddleOCR：开源免费，但对APP截图识别率0%
- GPT-4o Vision：付费API（0.03元/张），但准确率100%

**成本收益分析**：
```
方案对比：
PaddleOCR：
- 成本：0元（开源）
- 准确率：0%（无法使用）
- 代码量：需要预处理、后处理（约500行）

GPT-4o Vision：
- 成本：0.03元/张（用户付费19.9元/月可覆盖）
- 准确率：100%（72/72验证）
- 代码量：161行（services/vision_ocr_service.py）

结论：用少量API成本换取产品可用性
```

### 架构决策2: Prompt Engineering多行识别突破

**问题诊断过程**（2025-11-15）：
```
第一次测试：
- 输入：bill_dining.png（4笔交易）
- 输出：只识别出第一笔
- 假设1：token限制？ → 排除（图片token充足）
- 假设2：温度参数？ → 排除（temp=0.0已是最确定）
- 假设3：LLM理解问题？ → 确认！
```

**解决方案设计**：
改变LLM的思考流程，不是一次性提取，而是：
1. 先统计图片中有多少笔交易（几行独立金额）
2. 然后逐行提取详细信息
3. 最后验证：transactions数组长度 = transaction_count

**关键代码片段**：
```python
# services/vision_ocr_service.py:220-223
prompt = """
分析这张账单图片，提取所有交易记录。

重要：
1. 先统计图片中有多少笔交易（有几行独立金额就有几笔交易）
2. 然后逐行提取每一笔的详细信息
3. 确保 transactions 数组长度 = transaction_count

返回JSON格式：
{
  "transaction_count": 实际交易笔数,
  "transactions": [...]
}
"""
```

**验证结果**：
```
测试集：10张真实账单（72笔交易）
优化前：22/72识别成功（30%）
优化后：72/72识别成功（100%）

典型case:
- bill_dining.png: 1/4 → 4/4
- bill_shopping.png: 1/3 → 3/3
- real/4.png: 1/4 → 4/4（真实支付宝截图）
```

### 架构决策3: XAI混合架构（规则引擎 + LLM）

**设计原则**：
- 规则引擎负责决策逻辑（保证合规、可审计）
- LLM负责解释翻译（提升用户理解）

**实现细节**：
```python
# services/recommendation_service.py

# Step 1: 规则引擎计算
risk_score = calculate_risk_score(user_profile)
asset_allocation = get_allocation_strategy(risk_score, goal_period)

# Step 2: LLM翻译
explanation_prompt = f"""
用户风险偏好：{risk_profile}
目标期限：{goal_period}年
推荐配置：{asset_allocation}

请用通俗语言解释：
1. 为什么这个配置适合用户
2. 具体的收益测算
3. 风险提示
"""

explanation = llm.generate(explanation_prompt)

# Step 3: 返回结果 + 解释
return {
    "recommendation": asset_allocation,
    "explanation": explanation,
    "rule_params": {  # 可审计参数
        "risk_score": risk_score,
        "allocation_logic": "稳健型2年期策略"
    }
}
```

**核心优势**：
1. 决策逻辑透明：规则引擎的每一步都可追溯
2. 监管友好：满足金融推荐的合规要求
3. 用户理解：技术术语翻译成自然语言

### 问题1: "GPT-4o API成本会不会太高？"

**回答策略**（诚实+有深度）：
"当前成本是0.03元/张，对于C端用户，19.9元/月的订阅费可以覆盖600张识别成本。关键是先验证产品价值：用户愿不愿意为'3秒自动识别'付费。

如果用户规模达到10万级别，我们会考虑两个优化方向：
1. 短期：批量采购API，降低单价
2. 长期：自建模型微调，用开源Vision LLM（如LLaVA）替代GPT-4o

但现在MVP阶段，产品可用性比成本优化更重要。"

### 问题2: "如何应对竞品模仿？"

**回答策略**（强调技术壁垒）：
"技术壁垒在于Prompt Engineering的know-how。从30%到100%的识别率，我们花了10天迭代、测试了50+种prompt策略。竞品即使用GPT-4o，也需要重复这个试错过程。

而且我们不开源核心prompt设计，竞品看到的只是结果（100%识别率），看不到我们如何让LLM'先统计、再提取'的思考流程。

此外，XAI架构设计（规则引擎 + LLM的最优平衡点）也是我们的护城河。这不是简单调用API，而是系统工程能力。"

### 问题3: "用户数据隐私如何保障？"

**回答策略**（突出差异化优势）：
"这恰恰是我们相比支付宝、银行APP的核心优势：

1. **无需授权银行账户**：用户只上传截图，不需要绑定银行卡
2. **数据本地处理**：OCR识别后，数据存储在用户本地（Session State）
3. **可选导出备份**：用户可以导出JSON文件自行保管
4. **未来私有云部署**：To B客户可以部署私有云版本，数据完全内网隔离

对比：
- 支付宝记账：需要授权支付宝账户（高风险）
- 银行APP：数据存储在银行服务器（用户无控制权）
- WeFinance：数据由用户掌控（隐私优先）"

### 问题4: "异常检测的准确率如何？会不会误报太多？"

**回答策略**（诚实承认不足+优化方向）：
"现在用的是Z-score统计学方法，基于历史消费均值和标准差判断异常。坦白说，我们还没有做大规模的F1分数测试，这是下一步优化的重点。

目前的策略是：
1. **用户反馈学习**：用户可以标记'正常'或'欺诈'，系统会学习优化
2. **白名单机制**：用户可以添加信任商户，减少误报
3. **多维度检测**：不只看金额，还结合时间、地点、商户类型

下一阶段（1-3个月）的优化方向：
- 从统计学到机器学习（Random Forest或XGBoost）
- F1分数测试，目标准确率>0.85
- 引入图模型，分析消费行为序列

现阶段我们更强调'可控的误报'：宁可多提醒几次，让用户确认，也不要漏掉真正的风险。"

### 问题5: "为什么不直接对接支付宝/微信API，而是用截图识别？"

**回答策略**（突出产品定位差异）：
"这是我们深思熟虑的产品定位选择：

1. **隐私优先**：对接API需要用户授权账户，很多用户不愿意
2. **技术门槛**：支付宝/微信的API接入门槛很高，个人开发者拿不到
3. **适用场景更广**：截图方案不仅支持支付宝/微信，还支持银行APP、信用卡账单、纸质账单拍照
4. **差异化竞争**：API路线支付宝自己在做，我们做不过他们；截图识别是我们的独特优势

换个角度：我们的核心价值不是'数据获取'，而是'数据理解'（XAI推荐）和'风险预警'（异常检测）。截图OCR只是入口，不是终点。"

### 问题6: "10天开发是不是太短了？功能会不会不完善？"

**回答策略**（强调MVP理念+快速迭代）：
"10天确实很紧凑，但这恰恰体现了我们的MVP（最小可行产品）理念：

**我们10天验证的核心假设**：
1. Vision LLM能否100%识别账单截图？→ 验证通过（72/72）
2. Prompt Engineering能否解决多行识别？→ 验证通过（30%→100%）
3. XAI架构是否能让用户理解推荐？→ 需要进一步测试

**我们刻意没做的功能**：
- 数据持久化（用Session State足够验证核心流程）
- 用户系统（现阶段不需要）
- 精美UI（Streamlit够用，快速迭代）

**我们的优势是快速验证、快速迭代**：
- 发现问题 → 24小时内修复 → 重新部署
- 例如：11-15发现多行识别问题 → 当天优化prompt → 成功率100%

Lean Startup理论：先验证核心价值，再打磨完善度。"
